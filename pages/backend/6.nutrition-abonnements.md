## Tiers disponibles

| Tier | Description |
|---|---|
| `Free` | Accès limité — découverte du produit |
| `Pro` | Accès étendu — usage quotidien complet |
| `Business` | Accès illimité — usage intensif |

**Modélisation domaine :** `User.SubscriptionTier` → enum `SubscriptionTier { Free, Pro, Business }`
**Source de vérité :** base de données (pas le JWT).
**Rôles Keycloak :** `user` (tous les tiers) · `admin` (gestion interne — templates, RGPD)

## Règles & remarques

### Invariants domaine (non négociables — ne pas lever par tier)

- **Diet active = 1 par utilisateur** — règle domaine (409 si déjà active). Ce n'est pas un choix commercial, c'est un invariant du modèle. Lever cette limite casserait le calcul du bilan nutritionnel.
- **Export RGPD** — obligatoire pour tous les tiers sans exception (Art. 20 RGPD).
- **Suppression de compte** — idem (Art. 17 RGPD).

### Décisions produit

- **Templates en Free = ❌** — levier d'acquisition vers Pro. Les templates sont la première valeur ajoutée visible du passage à Pro.
- **Bilan limité à 7 jours en Free** — donne une aperçu fonctionnel sans remplacer l'usage Pro.
- **Les limites numériques** (2 / 20 / illimité etc.) sont des valeurs initiales — à ajuster selon les métriques d'usage réelles.

### Application dans le code

Les vérifications de tier se font dans la **couche Application** (services), jamais dans le domaine ni dans l'API :
```
UserService / DietPlanService / etc.
  → lire User.SubscriptionTier
  → comparer avec les limites du tier
  → lever une exception métier si dépassement (→ 403 Forbidden)
```
