## 2. Spécifications fonctionnelles

### 2.1 Fonctionnalités principales

#### Gestion des utilisateurs

- Inscription et authentification des utilisateurs via Keycloak
- Création et modification de profil utilisateur (données personnelles, objectifs)
- Gestion des préférences alimentaires et allergies

#### Calcul nutritionnel

- Calcul du métabolisme de base (BMR) selon différentes formules (Harris-Benedict, Mifflin-St Jeor)
- Détermination des besoins caloriques quotidiens en fonction du niveau d'activité
- Ajustement des besoins selon les objectifs (perte de poids, prise de masse, maintien)
- Répartition des macronutriments selon différents modèles de régimes

#### Gestion du régime alimentaire

- Création de plans alimentaires personnalisés
- Utilisation d'une API externe pour les données alimentaires
- Composition de repas et calcul automatique des apports nutritionnels
- Suivi quotidien des apports par rapport aux objectifs

#### Analyse et recommandations

- Visualisation des tendances d'apports nutritionnels
- Recommandations d'aliments en fonction des objectifs et préférences
- Alertes en cas de déséquilibre nutritionnel

#### Bilan nutritionnel par Diet

Le bilan est calculé à la demande pour une Diet donnée (`GET /diets/{id}/bilan`). Il couvre la période `Diet.StartDate` → `Diet.EndDate` (ou aujourd'hui si le régime est encore actif).

**Données retournées et affichages possibles :**

| Donnée                                        | Affichage interface                                    |
| --------------------------------------------- | ------------------------------------------------------ |
| Calories réelles par jour + `CalorieTarget`   | Courbe calories réelles vs ligne cible                 |
| Macros réelles par jour + `MacroDistribution` | Courbes protéines / glucides / lipides réels vs cibles |
| % d'atteinte journalier (calories + macros)   | Jauge ou barre de progression par jour                 |
| `WeightEntry` sur la période + `TargetWeight` | Courbe poids réel vs ligne poids cible                 |
| Moyennes journalières sur la période          | Résumé global du bilan                                 |
| Totaux sur la période                         | Récapitulatif consolidé                                |

**Règles :**

- Un seul endpoint agrège toutes les données — le frontend choisit les courbes à afficher
- Aucune entité `SavedBilan` — le bilan est toujours recalculé depuis les données existantes
- Si aucun repas n'est enregistré sur un jour, ce jour apparaît avec des valeurs à zéro

### 2.2 Parcours utilisateur

#### Création de compte et profil

1. Depuis l’interface frontend, l’utilisateur clique sur "Connexion".
2. Il est redirigé vers l’interface d’authentification de Keycloak :
   - S’il possède déjà un compte, il entre ses identifiants pour se connecter.
   - S’il n’a pas encore de compte, il sélectionne "S’enregistrer" pour créer un nouveau compte.
3. Après l’authentification ou l’enregistrement, l’utilisateur est redirigé automatiquement vers l’application frontend.
4. À sa première connexion, une requête est envoyée à l’API métier (.NET) afin de créer ou compléter le profil utilisateur avec les informations suivantes :
   - Âge, sexe, taille, poids
   - Niveau d’activité physique
   - Objectifs (perte de poids, maintien, prise de masse)
   - Préférences alimentaires et allergies
5. Le système effectue les calculs nutritionnels initiaux (BMR, TDEE, etc.) afin de personnaliser les recommandations.

👉 \_Pour une description technique détaillée du flux d’authentification, voir la section 3.3 Gestion de l'authentification.

#### Création d'un régime alimentaire

1. L'utilisateur sélectionne un type de régime (équilibré, cétogène, méditerranéen, etc.)
2. Le système propose une répartition de macronutriments adaptée
3. L'utilisateur peut ajuster cette répartition selon ses préférences
4. Le système génère des recommandations de repas conformes au régime choisi

#### Suivi quotidien

1. L'utilisateur recherche des aliments via l'API externe intégrée
2. L'utilisateur enregistre ses repas quotidiens
3. Le système calcule les apports nutritionnels et les compare aux objectifs
4. L'utilisateur visualise son bilan journalier et ses tendances hebdomadaires

### 2.3 Interfaces utilisateur et interopérabilité

#### Utilisation sans frontend

L'API est conçue pour être entièrement fonctionnelle sans nécessiter de frontend spécifique. Elle peut être utilisée de plusieurs façons :

- **Consommation directe par des services tiers** via les endpoints RESTful
- **Intégration avec des applications IoT** pour les appareils connectés de suivi de santé
- **Utilisation avec des chatbots** ou assistants vocaux
- **Intégration avec des applications tierces** via webhooks
- **Utilisation dans des environnements d'automatisation** (ex: Zapier, IFTTT)
- **Développement de scripts clients** avec tout langage supportant HTTP

#### Applications frontend prévues

Bien que l'API puisse fonctionner de manière autonome, une interface frontend est prévue (web, mobile, ou autre) :

- **Application frontend**
  - Interface utilisateur réactive et moderne
  - Tableaux de bord interactifs
  - Visualisation des données nutritionnelles
  - Interface d'administration

#### Autres interfaces potentielles

- **Application mobile** (iOS/Android)
- **Extension navigateur** pour suivi rapide
- **Interfaces pour wearables** (montres connectées, fitness trackers)
