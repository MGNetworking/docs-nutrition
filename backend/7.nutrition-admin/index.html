    Back-office Admin — API Nutrition¶

    Fonctionnalités réservées au rôle admin Keycloak. Dernière mise à jour : 2026-05-02

    Objectif¶

    Fournir à l'administrateur une vue d'ensemble de l'activité de l'application et la capacité de gérer le contenu partagé accessible à tous les utilisateurs.

    L'admin ne gère jamais les données personnelles d'un utilisateur individuel.

    Authentification & accès¶

    • Rôle Keycloak requis : admin (claim realm_access.roles)
    • Tous les endpoints /admin/** retournent 403 Forbidden si le rôle est absent
    • Le rôle admin est distinct du service account RGPD (nutrition-api-service) — voir infrastructure-keycloak-admin.md

    Principe RGPD — données agrégées uniquement¶

    Autorisé Interdit
    Compter le nombre d'utilisateurs par tier Lire le profil d'un utilisateur individuel
    Voir le nombre de diets actives Accéder aux repas ou pesées d'un utilisateur
    Voir le statut des jobs système Identifier qui est en grace period
    Créer du contenu partagé (templates) Modifier les données d'un utilisateur

    Endpoints¶

    GET /admin/dashboard¶

    Vue consolidée des KPIs de l'application.

    Réponse 200 OK : json { "users": { "total": 1240, "byTier": { "free": 980, "pro": 220, "business": 40 }, "newLast7Days": 34 }, "activity": { "activeDiets": 187, "mealsLast7Days": 4312, "accountsInGracePeriod": 3 } }

    GET /admin/system/health¶

    Statut opérationnel des jobs planifiés et du catalogue alimentaire.

    Implémentation : les données lastRunAt et status sont lues depuis les tables Hangfire ([HangFire].[Job], [HangFire].[State]) — aucune entité domaine nécessaire. Voir docs/pages/backend/annexes/infrastructure-hangfire.md.

    Réponse 200 OK : json { "offImport": { "lastRunAt": "2026-05-02T03:00:00Z", "status": "success", "foodItemsCount": 3247891 }, "rgpdPurge": { "lastRunAt": "2026-05-02T03:30:00Z", "status": "success", "purgedCount": 1 } }

    Statuts possibles : success · failed · never_run

    Contenu partagé — DietPlan templates¶

    Méthode Endpoint Description
    GET /admin/diet-plans/templates Lister tous les templates
    POST /admin/diet-plans/templates Créer un template
    PUT /admin/diet-plans/templates/{id} Modifier un template
    DELETE /admin/diet-plans/templates/{id} Supprimer un template

    Note : la lecture des templates par les utilisateurs (Pro/Business) se fait via GET /diet-plans/templates — pas via les endpoints admin.

    Ce que l'admin ne peut pas faire¶

    • Lire, modifier ou supprimer les données d'un utilisateur individuel
    • Changer le SubscriptionTier d'un utilisateur (opération future — hors scope v1)
    • Voir les données personnelles (repas, pesées, plans personnels)
