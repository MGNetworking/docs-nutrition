    Specs techniques

    3. Spécifications techniques¶

    3.1 Architecture globale¶

    Le système sera implémenté comme une application monolithique déployée sur Kubernetes, qui interagira avec Keycloak pour l'authentification et une API externe pour les données alimentaires.

    3.2 Responsabilités du service¶

    L'API Nutrition est conçue comme un service monolithique qui gère l'ensemble des fonctionnalités :

    Gestion des utilisateurs¶

    • Création et gestion des comptes utilisateurs (inscription, modification de profil)
    • Interface avec Keycloak pour l'authentification et l'autorisation
    • Gestion du profil utilisateur (données physiologiques, préférences, objectifs)
    • Calcul du métabolisme de base (BMR) et des besoins caloriques
    • Gestion des identifiants et tokens pour l'API alimentaire au nom de l'utilisateur

    Gestion du régime alimentaire¶

    • Création et gestion des régimes personnalisés (un seul actif à la fois par utilisateur)
    • Intégration avec l'API Open Food Facts pour les données nutritionnelles
    • Planification des repas
    • Calcul des apports nutritionnels
    • Génération de recommandations
    • Analyse des tendances nutritionnelles

    Gestion des repas¶

    • Enregistrement des consommations alimentaires
    • Recherche d'aliments via l'API externe (avec mise en cache local)
    • Gestion de l'historique des repas
    • Calcul des apports nutritionnels par repas
    • Vérification optionnelle de la conformité avec le régime actif

    3.3 Gestion de l'authentification¶

    L'authentification sera gérée par Keycloak, un serveur d'identité open-source.

    Flux d'authentification¶

    1. Authentification via l'application frontend
    2. L'utilisateur déclenche le processus de connexion depuis l'interface web
    3. Il est redirigé vers la page d'authentification centralisée Keycloak
    4. S'il possède un compte, il se connecte ; sinon, il crée un compte

    5. Une fois authentifié, il est redirigé vers l'application frontend avec un token JWT

    6. Enregistrement côté API métier

    7. À la première connexion, l'application transmet les informations d'identification à l'API
    8. L'API crée un profil utilisateur interne lié à l'identifiant Keycloak

    9. Ce lien permet d'associer l'identité Keycloak aux données métier de l'utilisateur

    10. Contrôle des accès aux fonctionnalités

    11. Pour chaque requête, un token JWT est transmis dans l'en-tête Authorization
    12. L'API vérifie la validité du token (signature, expiration, issuer)

    13. Les droits d'accès sont vérifiés selon les rôles associés (User, Admin, Nutritionist)

    14. Accès à des services externes

    15. L'API interagit avec Open Food Facts de manière transparente pour l'utilisateur
    16. Aucune authentification requise pour Open Food Facts (API publique)

    Pour une description visuelle du flux, voir l'annexe workflow_Flux_authentification.mermaid

    Implémentation de la sécurité¶

    La sécurité est configurée dans Program.cs via les middlewares ASP.NET Core :

    • AddAuthentication().AddJwtBearer() pour la validation des tokens Keycloak
    • AddAuthorization() avec des policies basées sur les rôles
    • Les endpoints sont protégés via les attributs [Authorize] et [Authorize(Roles = "...")]

    3.4 Paradigmes de développement¶

    Domain-Driven Design (DDD)¶

    Le projet est structuré selon les principes du Domain-Driven Design — le code s'organise autour du métier, pas autour de la base de données.

    Concept Rôle dans ce projet
    Aggregate Root Point d'entrée du domaine — User, Diet, DietPlan, Meal, FoodItem
    Entité enfant Vit uniquement via son parent — MealItem, WeightEntry, SavedFoodItem
    Value Object Groupe de valeurs immuables sans identité — MacroDistribution, NutritionInfo, NutritionNeeds
    Domain Service Règle métier impliquant plusieurs agrégats — ex: vérification Diet active
    Invariant Règle toujours vraie, validée dans le constructeur — ex: somme macros = 100%

    L'architecture cible est une architecture 4 couches (Diamant) où les dépendances convergent vers le Domain. Voir section "Structure du projet" ci-dessous.

    Test-Driven Development (TDD)¶

    Le développement suit le cycle Red → Green → Refactor, appliqué couche par couche dans l'ordre d'implémentation.

    Cycle par fonctionnalité :

    1. Red — écrire un test qui échoue décrivant le comportement attendu
    2. Green — écrire le minimum de code pour faire passer le test
    3. Refactor — nettoyer sans changer le comportement (le test reste vert)

    Ordre d'application par couche :

    Ordre Couche Ce qu'on teste en TDD Outils
    1 Domain Invariants, constructeurs, Value Objects, règles métier pures xUnit + assertions
    2 Application Use cases, orchestration, réponses aux cas d'erreur xUnit + Moq (mock repositories)
    3 Infrastructure Persistance EF Core, requêtes, migrations xUnit + Testcontainers
    4 API Contrats HTTP, codes de retour, format des réponses WebApplicationFactory

    Convention de nommage des tests :

    [Méthode_ou_Comportement]_[Contexte]_[Résultat_attendu]

Exemples :
CreateDiet_WhenNoWeightEntry_Throws422
MacroDistribution_WhenSumIsNot100_ThrowsDomainException
LaunchDiet_WhenDietAlreadyActive_Returns409

    Règle structurelle : chaque projet de test correspond à une couche.

    NutritionApi.Domain.Tests/          ← tests unitaires Domain
NutritionApi.Application.Tests/     ← tests unitaires Application (Moq)
NutritionApi.Infrastructure.Tests/  ← tests d'intégration (Testcontainers)
NutritionApi.Api.Tests/             ← tests API end-to-end (WebApplicationFactory)

    Stratégie de couverture de code¶

    La couverture est mesurée avec coverlet.collector (déjà installé dans les 3 projets de test) et visualisée avec ReportGenerator.

    Seuils minimum par couche :

    Couche Seuil Métrique prioritaire
    Domain 90 % Branch coverage — chaque invariant testé dans les deux sens
    Application 80 % Line coverage
    Infrastructure 70 % Line coverage
    API 70 % Line coverage

    Lancer les tests avec couverture :

    dotnet test --settings tests/coverage.runsettings --collect:"XPlat Code Coverage" --results-directory ./coverage

    Générer le rapport HTML :

    # Installer ReportGenerator (une seule fois, outil global)
dotnet tool install -g dotnet-reportgenerator-globaltool

# Générer le rapport
reportgenerator -reports:"coverage/**/coverage.cobertura.xml" -targetdir:"coverage/report" -reporttypes:Html

    Le fichier tests/coverage.runsettings est à la racine du dossier tests/ — il configure le format Cobertura et exclut les projets de tests eux-mêmes de la mesure.

    3.6 API Endpoints¶

    L'API est organisée selon les principes REST :

    Authentification (via Keycloak)¶

    POST   /api/v1/auth/register       // Inscription d'un utilisateur
POST   /api/v1/auth/login          // Connexion et génération de token JWT
POST   /api/v1/auth/refresh        // Rafraîchissement du token
POST   /api/v1/auth/logout         // Déconnexion

    Utilisateurs¶

    GET    /api/v1/users/me                  // Récupération du profil utilisateur
PUT    /api/v1/users/me                  // Mise à jour du profil utilisateur
POST   /api/v1/users/me/preferences      // Définition des préférences alimentaires
GET    /api/v1/users/me/stats            // Statistiques de l'utilisateur

    Aliments (via Open Food Facts + cache local)¶

    GET    /api/v1/foods/search              // Recherche d'aliments
GET    /api/v1/foods/{externalId}        // Détails d'un aliment

    Régimes alimentaires¶

    GET    /api/v1/diets                     // Liste des régimes de l'utilisateur
POST   /api/v1/diets                     // Création d'un régime
GET    /api/v1/diets/{id}                // Détails d'un régime
PUT    /api/v1/diets/{id}                // Mise à jour d'un régime
DELETE /api/v1/diets/{id}                // Suppression d'un régime
PUT    /api/v1/diets/{id}/activate       // Activation d'un régime (désactive les autres)

    Nutrition¶

    GET    /api/v1/nutrition/calculate       // Calcul des besoins nutritionnels (BMR, TDEE)
GET    /api/v1/nutrition/daily           // Bilan nutritionnel journalier
GET    /api/v1/nutrition/weekly          // Bilan nutritionnel hebdomadaire
GET    /api/v1/nutrition/recommendations // Recommandations alimentaires

    Repas¶

    GET    /api/v1/meals                     // Liste des repas (filtrable par date)
POST   /api/v1/meals                     // Enregistrement d'un repas
GET    /api/v1/meals/{id}                // Détails d'un repas
PUT    /api/v1/meals/{id}                // Mise à jour d'un repas
DELETE /api/v1/meals/{id}                // Suppression d'un repas

    Format de données¶

    Toutes les requêtes et réponses utilisent le format JSON. Exemple de réponse pour le calcul des besoins nutritionnels :

    {
  "bmr": 1762.5,
  "tdee": 2467.5,
  "targetCalories": 1967.5,
  "macroDistribution": {
    "proteins": { "percentage": 25, "grams": 123, "calories": 492 },
    "carbs": { "percentage": 45, "grams": 221, "calories": 884 },
    "fats": { "percentage": 30, "grams": 65, "calories": 585 }
  },
  "goal": "WeightLoss",
  "activityLevel": "ModeratelyActive"
}

    3.7 Performances et scalabilité¶

    Objectifs de performance¶

    • Temps de réponse moyen < 100ms pour les requêtes simples
    • Temps de réponse moyen < 500ms pour les calculs complexes
    • Capacité à gérer 200 requêtes/seconde
    • Disponibilité > 99.9%

    Stratégies d'optimisation¶

    • Pagination pour les listes volumineuses
    • Mise en cache avec IMemoryCache (Caffeine équivalent .NET)
    • Exécution asynchrone (async/await) pour toutes les opérations I/O
    • Circuit breaker Polly pour les appels vers Open Food Facts
    • Pool de connexions à la base de données via EF Core / Npgsql

    Scalabilité¶

    • Architecture sans état (stateless) — scaling horizontal facilité
    • Auto-scaling des pods via Kubernetes HPA (CPU/mémoire)
    • Rate limiting via middleware ASP.NET Core
    • Requêtes EF Core optimisées (pas de SELECT *, projections, index)

    3.8 Sécurité¶

    Authentification et autorisation¶

    • Authentification via Keycloak avec support OAuth2/OIDC
    • Support de fournisseurs externes (Google, Facebook) via Keycloak
    • Gestion des rôles : User, Admin, Nutritionist
    • Validation des tokens JWT côté serveur à chaque requête
    • Vérification des permissions par endpoint via [Authorize(Roles = "...")]

    Protection des données¶

    • Chiffrement des données sensibles au repos
    • HTTPS/TLS pour toutes les communications
    • Validation des entrées via Data Annotations et FluentValidation
    • Protection contre les injections SQL via EF Core (requêtes paramétrées)
    • Journalisation des accès et opérations sensibles

    Sécurité réseau¶

    • Réseau privé Kubernetes pour isoler les composants internes
    • Network Policies Kubernetes pour contrôler le trafic inter-pods
    • WAF (Web Application Firewall) pour la protection contre les attaques communes

    Pipeline CI/CD (GitHub Actions)¶

    Le pipeline automatise les étapes suivantes à chaque push :

    1. Build — dotnet build
    2. Tests — dotnet test (unitaires + intégration)
    3. Analyse qualité — SonarQube
    4. Build image Docker — docker build
    5. Push image — vers le registry
    6. Déploiement — kubectl apply vers l'environnement cible
